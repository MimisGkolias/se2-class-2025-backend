## 🟢 Lab 3 — Express / Node backend (βασικά)

**Στόχος:** να ανοίγω οποιοδήποτε backend και να βρίσκω αμέσως → *πού ορίζεται το route, τι input παίρνει, τι επιστρέφει*. (Θεμέλιο για AVA tests.)

### Τρέξιμο project

```bash
npm install        # κατεβάζει ό,τι λέει το package.json
npm run dev        # nodemon (auto-reload) — ή: npm start
# port από .env (π.χ. PORT=3000) → http://localhost:3000/
```

### Express skeleton

```jsx
import express from "express";
import "dotenv/config";
const app = express();
const PORT = process.env.PORT || 3000;
app.use(express.json());   // ⚠️ ΧΩΡΙΣ αυτό το req.body είναι undefined (POST)
app.get("/", (req, res) => res.send("Hello world!"));
app.listen(PORT, () => console.log(`running on ${PORT}`));
```

### Πώς διαβάζω input & πώς απαντά

| Τι | Κώδικας |
| --- | --- |
| URL param | `app.get('/users/:id', (req,res)=>{ req.params.id })` |
| JSON body (POST) | `req.body.name` (χρειάζεται `express.json()`) |
| Query string | `req.query.foo` |
| Απάντηση | `res.send(...)` ή `res.json(...)` → default status **200** |
| Catch-all / 404 | `app.all('*', ...)` |

### Middleware & `next()` (πέφτει & σε θεωρία)

> Middleware = function με `(req, res, next)`. Κανόνας: στέλνει απάντηση με `res.send` **ή** καλεί `next()` — αλλιώς το request «κρεμάει».
> 

### Async route + error handling

```jsx
app.get("/weather/:city", async (req, res, next) => {
	try {
		const resp = await axios.get(url);
		res.send(resp.data);
	} catch (err) { next(err); }
});
```

### 📌 HTTP status codes (για tests & θεωρία)

| Code | Σημασία | Πότε |
| --- | --- | --- |
| **200** | OK | GET / PUT επιτυχία |
| **201** | Created | POST (νέο) |
| **204** | No Content | DELETE (κενό body) |
| **400 / 401 / 403 / 404** | Bad Request / Unauthorized / Forbidden / Not Found | λάθη πελάτη |
| **500** | Internal Server Error | λάθος server |

<aside>
⚠️

**Παγίδα swagger-generated server (2024):** ο auto-generated stub επιστρέφει **200 ΠΑΝΤΟΥ** (ακόμα & σε POST/DELETE), γιατί ο `writer.js` κάνει default σε 200. Άρα στα tests του stub βάζε `t.is(res.status, 200)`, όχι 201/204.

</aside>

## ⚛️ Lab 4 — React frontend

**Στόχος:** να ανοίγω React frontend και να βρίσκω αμέσως κουμπιά / inputs / κείμενα → selectors για **Cypress tests**.

### Τρέξιμο

```bash
npm install
npm start          # react-scripts → http://localhost:3000
```

### Δομή

- `src/index.js` → μοντάρει το `<App />` στο `#root`.
- `src/App.js` → το component με όλο το UI (εδώ ζει το JSX που τεστάρεις).

### React έννοιες (τι κάνουν)

| Έννοια | Στον κώδικα | Τι κάνει |
| --- | --- | --- |
| **Component** | `function App()` επιστρέφει JSX | Η «οθόνη» που φορτώνει ο browser |
| **JSX** | `className` (όχι `class`), `{expr}` για JS | Το markup που render-άρεται |
| **`useState`** | `const [val, setVal] = useState("")` | State που αλλάζει με input/click |
| **`useEffect`** | `useEffect(()=>{...}, [val])` | Τρέχει όταν αλλάζει το dependency |
| **Events** | `onChange`, `onClick` | Οι ενέργειες που προσομοιώνει το Cypress |
| **Controlled input** | `value={val}` δεμένο σε state | Γι' αυτό το `.type()` αλλάζει το char count |
| **MUI** | `<Button>`, `<Grid>` | Render-άρονται ως κανονικά `<button>`/`<div>` |

### 🎯 Πώς βρίσκω selectors για Cypress

| Στοιχείο UI | Cypress |
| --- | --- |
| Επικεφαλίδα | `cy.contains('h1', 'Welcome to React Demo')` |
| Input | `cy.get('input[placeholder="Type something..."]')` |
| Κουμπί Submit | `cy.contains('button', 'Submit').click()` |
| Κείμενο count | `cy.contains('Character Count: 5')` |

### `cy.contains()` — 2 μορφές

- `cy.contains('κείμενο')` → πιάνει το στοιχείο που περιέχει το κείμενο (όταν είναι **μοναδικό**).
- `cy.contains('selector', 'κείμενο')` → περιορίζει σε συγκεκριμένο element (π.χ. `'button'`, `'p'`).
- **Βάζω tag όταν:** (α) το κείμενο υπάρχει σε πολλά σημεία → disambiguation, ή (β) θα κάνω **click** σε συγκεκριμένο element (το MUI `<Button>` φτιάχνει nested `<span>` → το `'button'` πιάνει το πραγματικό `<button>`).

> Tip: προτίμα **text / placeholder / role**· απόφυγε αυτο-παραγόμενες MUI κλάσεις (`.css-...`).

## 🔁 Lab 5 — CI/CD (GitHub Actions) *[contingency]*

**Τι διδάσκει:** μια αλλαγή κώδικα περνά αυτόματα από **CI** (build + tests) και, αν περάσει, **CD** (deploy). Όλα σε ένα αρχείο: `.github/workflows/ci.yml`.

### Ανατομία `ci.yml`

```yaml
name: CI
on:                          # ΠΟΤΕ τρέχει (triggers)
  push:
    branches: [ main ]       # push στο main
  pull_request:              # κάθε PR
  workflow_dispatch:         # χειροκίνητα
jobs:
  ci:                        # JOB 1: build & test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '22' }
      - run: npm install
      - run: npm test            # tests (ava)
  cd:                        # JOB 2: deploy
    needs: ci                # ΜΟΝΟ αν περάσει το ci
    if: github.ref == 'refs/heads/main'   # ΜΟΝΟ στο main
    steps:
      - uses: johnbeynon/render-deploy-action@v0.0.8
```

### 5 έννοιες-κλειδιά

| Όρος | Τι κάνει |
| --- | --- |
| `on:` | Triggers: πότε ξεκινά (push / PR / manual) |
| `jobs:` | Ανεξάρτητες εργασίες (`ci`, `cd`) |
| `steps:` | Διαδοχικά βήματα μέσα σε job |
| `needs: ci` | Εξάρτηση → το `cd` δεν τρέχει αν αποτύχει το `ci` |
| `if: ...main` | Conditional → deploy μόνο στο `main` |

### Backend vs Frontend pipeline

- **Backend:** test step = `npm test` → **AVA**.
- **Frontend:** test step = **Cypress** (`cypress-io/github-action@v6`) με `start: npm start` + `wait-on: http://localhost:3000`.
- Ίδιος σκελετός, αλλάζει μόνο το **πώς τεστάρει**.

### 🎯 Πιθανές ασκήσεις εξέτασης

**Θεωρία (Lec 4):** οφέλη CI/CD · CI vs CD · στάδια pipeline (checkout → install → test → deploy).

**Micro-task (YAML):**

- «τρέξε τα tests» → `- run: npm test`
- «deploy μόνο στο main» → `if: github.ref == 'refs/heads/main'`
- «cd να εξαρτάται από ci» → `needs: ci`
- «χειροκίνητο trigger» → `workflow_dispatch:`

## 🧪 Lab 6 — Backend Testing με AVA (⭐ το πιο σημαντικό)

Το setup (`test.before`) σου δίνεται **ΕΤΟΙΜΟ**. Εσύ γράφεις μόνο το σώμα του test: *χτύπα route → έλεγξε status → έλεγξε body*. Το path πάντα **χωρίς leading slash**.

### ⚙️ Setup boilerplate (σου δίνεται έτοιμο — καλό να το ξέρεις)

Σηκώνει τον `app` σε τυχαία θύρα πριν τα tests και τον κλείνει στο τέλος. Αλλάζει μόνο το πώς φτιάχνεις τον client (`got` vs `fetch`).

**🟠 με got:**

```jsx
import test from "ava";
import http from "node:http";
import got from "got";
import app from "../src/app.js";   // ο Express app (πρέπει να γίνεται export)

test.before(async (t) => {
  t.context.server = http.createServer(app);
  const server = t.context.server.listen();          // τυχαία ελεύθερη θύρα
  const { port } = server.address();
  t.context.got = got.extend({
    responseType: "json",
    prefixUrl: `http://localhost:${port}`,
  });
});

test.after.always((t) => {
  t.context.server.close();
});
```

**🔵 με fetch:** ίδια λογική, αλλά το `fetch` δεν έχει `extend`/`prefixUrl` → φτιάχνεις δικό σου wrapper με το base URL:

```jsx
import test from "ava";
import http from "node:http";
import app from "../src/app.js";

test.before(async (t) => {
  t.context.server = http.createServer(app);
  const server = t.context.server.listen();          // τυχαία ελεύθερη θύρα
  const { port } = server.address();
  const baseUrl = `http://localhost:${port}`;
  t.context.fetch = (path, opts) => fetch(`${baseUrl}/${path}`, opts);
});

test.after.always((t) => {
  t.context.server.close();
});
```

### 🔵 fetch dialect (εξέταση 2026) — GET

```jsx
test("GET /organizers/1", async (t) => {
  const res = await t.context.fetch("api/v1/organizers/1"); // ΧΩΡΙΣ leading slash ΑΛΛΑ ΜΕ το prefix του app
  t.is(res.status, 200);                                    // .status (ΟΧΙ statusCode)
  const body = await res.json();                            // ΠΡΕΠΕΙ await .json()
  t.is(body.data.organizationName, "Night Owl Productions"); // .data: ο server τυλίγει { success, message, data }
});
```

### 🔵 fetch dialect — POST (σταθερό boilerplate)

```jsx
test("POST /organizers", async (t) => {
  const res = await t.context.fetch("organizers", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ name: "Dimitris" }),
  });
  t.is(res.status, 201);                               // POST → 201
  const body = await res.json();
  t.is(body.status, "success");
});
```

### 🔴 fetch — Error route (το fetch ΔΕΝ πετάει)

Σε αντίθεση με το `got`, το `fetch` **ΠΟΤΕ** δεν πετάει exception σε 4xx/5xx — ο κωδικός μπαίνει απλώς στο `res.status`. Άρα τον ελέγχεις κατευθείαν, χωρίς `t.throwsAsync`:

```jsx
test("GET /api/data/secret χωρίς token → 401", async (t) => {
  const res = await t.context.fetch("api/data/secret");  // ΔΕΝ πετάει
  t.is(res.status, 401);                                 // ο κωδικός είναι εδώ
  const body = await res.json();
  t.is(body.error, "Unauthorized");
});
```

### 🟠 got dialect (Lab 6 sgarden)

```jsx
test("GET /api", async (t) => {
  const { body, statusCode } = await t.context.got("api");  // body ΗΔΗ JSON
  t.is(statusCode, 200);                                    // .statusCode
  t.is(body.message, "It works!");
});
// POST με got — το json γίνεται serialize αυτόματα (όχι JSON.stringify):
test("POST route με got", async (t) => {
  const { body, statusCode } = await t.context.got.post("route", {
    json: { name: "Dimitris" },
  });
  t.is(statusCode, 201);
  t.is(body.status, "success");
});
// ⚠️ got: σε 4xx/5xx ΠΕΤΑΕΙ exception (το fetch ΟΧΙ) → δες Κανόνα-ασπίδα
```

### 🔴 Error route (ο got πετάει σε 4xx/5xx)

Ο γενικός `got(...)` σε status 4xx/5xx **πετάει** — άρα το `t.is(statusCode, 401)` δεν φτάνει ποτέ. Δύο σωστοί τρόποι:

```jsx
// Α) πιάσε το exception → το status είναι στο err.response
test("GET /api/data/secret χωρίς token", async (t) => {
  const err = await t.throwsAsync(() => t.context.got("api/data/secret"));
  t.is(err.response.statusCode, 401);
});
// Β) πες στον got να ΜΗΝ πετάει
test("GET /api/data/secret → 401", async (t) => {
  const { statusCode } = await t.context.got("api/data/secret", {
    throwHttpErrors: false,
  });
  t.is(statusCode, 401);
});
```

### 🔁 PUT & DELETE (γενικές περιπτώσεις — έτοιμα)

**PUT** = ενημέρωση (στέλνεις body, status **200**) · **DELETE** = διαγραφή (συχνά **κενό body** → 200 ή 204).

**🟠 με got:**

```jsx
// PUT — json body (σεριαλίζεται μόνο του)
test("PUT /books/1", async (t) => {
  const { body, statusCode } = await t.context.got.put("books/1", {
    json: { title: "Updated", author_id: 1, category_id: 1, published_year: 2020 },
  });
  t.is(statusCode, 200);
  t.is(body.title, "Updated");
});
// DELETE — ⚠️ το body είναι ΣΥΧΝΑ ΚΕΝΟ → responseType:"text"
// (αλλιώς ο got προσπαθεί JSON.parse("") και ΣΚΑΕΙ)
test("DELETE /books/1", async (t) => {
  const { statusCode } = await t.context.got.delete("books/1", { responseType: "text" });
  t.is(statusCode, 200);   // ή 204 αν ο server γυρίζει No Content
});
```

**🔵 με fetch:**

```jsx
// PUT
test("PUT /books/1", async (t) => {
  const res = await t.context.fetch("books/1", {
    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ title: "Updated" }),
  });
  t.is(res.status, 200);
  const body = await res.json();
  t.is(body.title, "Updated");
});
// DELETE — το fetch ΔΕΝ σκάει σε κενό body
test("DELETE /books/1", async (t) => {
  const res = await t.context.fetch("books/1", { method: "DELETE" });
  t.is(res.status, 200);   // ή 204
});
```

<aside>
⚠️

**Η #1 παγίδα στο DELETE:** γυρίζει κενό body. Με `got` + `responseType:"json"` → σκάει «Unexpected end of JSON». Λύση: `responseType:"text"` ανά request. Το `fetch` δεν έχει αυτό το πρόβλημα.

</aside>

### 🛡️ Κανόνας-ασπίδα (κοίτα την 1η γραμμή του test)

- **fetch** ⇒ `res.status` + `await res.json()` + (POST: γράφεις headers & `JSON.stringify`)
- **got** ⇒ `res.statusCode` + body **ήδη** JSON + (POST: `{ json: {...} }`, χωρίς stringify)
- ⚠️ **got πετάει exception σε 4xx/5xx** (το fetch ΟΧΙ): για error route → `await t.throwsAsync(...)` ή `got(url, { throwHttpErrors: false })`. Το `responseType: "json"` πετάει αν το route γυρίζει σκέτο text.
- Η παραλλαγή του server (`app.listen(0)` / `test-listen`) **δεν σε αφορά** — σου δίνεται έτοιμη.

### 🧰 Assertions

```jsx
t.is(a, b)          // ===  (status, τιμές)
t.deepEqual(a, b)   // ολόκληρα objects/arrays ίσα
t.true(res.ok)      // status 200–299
t.truthy(x)         // υπάρχει / όχι null
t.regex(s, /.../)   // pattern
```

## 🖱️ Lab 7 — Frontend Testing με Cypress

**Στόχος:** E2E test — ο Cypress ανοίγει browser, κάνει κλικ/πληκτρολόγηση **σαν χρήστης** και ελέγχει το UI. Το app τρέχει στο `http://localhost:3000`.

### ▶️ Setup

```bash
npm install -D cypress
npx cypress open     # GUI (διάλεξε "E2E Testing") — ή headless: npx cypress run
```

Φτιάξε `cypress.config.js` με `baseUrl` και βάλε τα tests στο `cypress/e2e/*.cy.js`:

```jsx
const { defineConfig } = require("cypress");
module.exports = defineConfig({
  e2e: { baseUrl: "http://localhost:3000" },
});
```

### 📁 Πού μπαίνουν τα tests (δομή φακέλων)

Το `npx cypress open` φτιάχνει αυτόματα τη δομή. Τα δικά σου specs → `cypress/e2e/`:

```
cypress/
  e2e/
    init.cy.js          ← εδώ γράφεις τα tests (όνομα *.cy.js)
  support/
    e2e.js              ← τρέχει ΠΡΙΝ από κάθε spec (global setup)
    commands.js         ← custom εντολές (cy.login() κ.λπ.)
  fixtures/             ← mock δεδομένα (π.χ. events.json)
cypress.config.js
```

> **Στην εξέταση** συνήθως υπάρχει ΈΤΟΙΜΟ `cypress/e2e/template.cy.js` (όπως στο Thessevent) — απλώς γράφεις μέσα το `it(...)`. Το `support/e2e.js` (π.χ. `slowCypressDown`) είναι ήδη ρυθμισμένο — δεν το πειράζεις.
> 

### 🎯 Selectors — σειρά προτεραιότητας

2. **id** → `cy.get('#cart-count')`

### 🧰 Εντολές-κλειδιά — selectors & ενέργειες

| Εντολή | Τι κάνει | Παράδειγμα (juice-shop) |
| --- | --- | --- |
| `cy.visit(url)` | ανοίγει τη σελίδα — **υποχρεωτικό 1ο βήμα** | `cy.visit("http://localhost:3000")` |
| `cy.get("#id")` | 1 element με μοναδικό id | `cy.get("#cart-count")` |
| `cy.get(".class")` | ΌΛΑ τα elements με την κλάση (collection) | `cy.get(".product-card")` |
| `cy.get('[data-testid="x"]')` | element με data-testid | `cy.get('[data-testid="add-to-cart-1"]')` |
| `cy.contains("txt")` | element που περιέχει το κείμενο | `cy.contains("Juice Shop")` |
| `cy.contains("sel","txt")` | συγκεκριμένο element με κείμενο | `cy.contains("button","Add to Cart")` |
| `.click()` | κλικ | `cy.get("#view-cart").click()` |
| `.type("txt")` | πληκτρολόγηση σε `<input>` | `cy.get("#search-input").type("orange")` |
| `.clear()` | καθαρίζει το input | `cy.get("#search-input").clear()` |
| `.select("val")` | επιλογή σε `<select>` (dropdown) | `cy.get("#category-filter").select("vegetable")` |
| `.within(() => {…})` | περιορίζει queries ΜΕΣΑ σε element | `cy.get("#header").within(...)` |
| `.find(".x")` | ψάχνει απόγονο μέσα στο element | `.product-card").first().find(".add-to-cart-btn")` |
| `.first()` / `.eq(n)` | 1ο / n-οστό element σε collection | `cy.get(".product-card").eq(1)` |
| `.invoke("text")` | παίρνει το κείμενο για επεξεργασία | `...).invoke("text").then(Number)` |

### ✅ Assertions (`.should(...)`) — κάθε περίπτωση

| Assertion | Τι ελέγχει | Πότε το χρησιμοποιώ |
| --- | --- | --- |
| `.should("have.length", n)` | **πλήθος** elements σε collection | `.product-card").should("have.length", 4)` |
| `.should("have.length.greaterThan", n)` | πλήθος > n | category "mixed" → >1 |
| `.should("have.text", "s")` | **ΑΚΡΙΒΕΣ** κείμενο ενός element | `#product-count").should("have.text","4")` |
| `.should("contain", "s")` / `.should(”contain.text", "s")` | **περιέχει** κείμενο (όχι ακριβές) | `#empty-cart").should("contain.text","empty")` |
| `.should("exist")` | υπάρχει στο DOM | `.cart-item").should("exist")` |

| `routes/` | **Χάρτης URL → handler** — λέει «ποιο path πάει σε ποιον controller». Δεν έχει λογική. | 1 σχόλιο ανά group routes (π.χ. «/books routes») |
| `controllers/` | Δέχεται το request, τραβάει input (`req.params/body`), καλεί το **service**, στέλνει `res`. Ο «διαχειριστής» — όχι λογική. | JSDoc με **HTTP method + route** πάνω από κάθε handler |
| `services/` | **Η καρδιά** — εδώ ζει η business logic (υπολογισμοί, κανόνες, μιλάει με τα models) | JSDoc «τι κάνει + τι επιστρέφει» (`@param`/`@returns`) |
| `models/` | Ορίζει το **σχήμα των δεδομένων** (πεδία ενός Book/User) & μιλάει στη βάση (Sequelize/Mongoose) | file-header «model X» + σχόλιο σε μη-προφανή πεδία |
| `middleware/` | Functions `(req,res,next)` που τρέχουν **ανάμεσα** σε request & handler: auth, logging, error handling | JSDoc «τι ελέγχει/κάνει + πότε καλεί `next()`» |
| `utils/` / `helpers/` | Μικρές **επαναχρησιμοποιήσιμες** βοηθητικές (format, writeJson, hashing) — καθαρές, χωρίς state | JSDoc utility με `@param`/`@returns` |
| `validators/` | Ελέγχουν ότι το input είναι έγκυρο πριν φτάσει στη λογική | JSDoc «τι κανόνα επικυρώνει» |

## 🛰️ Lab 10 — DAST (Dynamic Application Security Testing) *[contingency]*

**Τι είναι DAST:** έλεγχος της εφαρμογής **ενώ τρέχει** (runtime) για ευπάθειες που ΔΕΝ φαίνονται στον στατικό κώδικα. "Black-box" — χτυπάς την εφαρμογή απ' έξω σαν επιτιθέμενος. Εργαλείο lab: **OWASP ZAP** + manual pentesting πάνω στο **Juice Shop**.

<aside>
🎯

**Καθαρά θεωρητικό** — δεν υπάρχει κώδικας/άσκηση, μόνο PDF. Στην εξέταση πέφτει ως **θεωρία**: Static vs Dynamic, ZAP scans, Pentest vs Vuln Assessment, XSS/SQLi/Path Traversal & πρόληψη.

</aside>

### 🔑 SAST vs DAST (η #1 ερώτηση)

| Άξονας | SAST (Lab 8) | DAST (Lab 10) |
| --- | --- | --- |
| Πότε | χωρίς εκτέλεση, στον πηγαίο κώδικα | σε runtime, στην εφαρμογή που τρέχει |
| Οπτική | White-box (βλέπεις κώδικα) | Black-box (σαν επιτιθέμενος) |
| Βρίσκει | code patterns (SQLi, secrets) | runtime/config (headers, auth, real exploits) |
| Εργαλεία | Semgrep, Cyclopt | OWASP ZAP, Burp, Nmap, Metasploit |

### 🛠️ ZAP — τύποι scan

| Scan | Τι κάνει |
| --- | --- |
| **Spider** | crawl URLs/links — **στατικό** περιεχόμενο, ΧΩΡΙΣ JS |
| **Spider Ajax** | τρέχει JS & προσομοιώνει browser → **δυναμικό** περιεχόμενο |
| **Active Scan** | στέλνει **crafted payloads** (SQLi, XSS) — επιθετικό |
| **Passive Scan** | αναλύει traffic ΧΩΡΙΣ extra requests (missing headers, weak SSL) |

### 🥷 Pentest vs Vulnerability Assessment

| Aspect | Penetration Testing | Vulnerability Assessment |
| --- | --- | --- |
| Στόχος | βρες ΚΑΙ **exploit** → δείξε impact | βρες & αξιολόγησε severity, ΧΩΡΙΣ exploit |
| Scope | στενό, βαθύ | ευρύ, επιφανειακό scan |
| Από ποιον | έμπειροι pentesters + tools | κυρίως automated tools |
| Outcome | attack narratives + remediation | λίστα ευπαθειών + risk levels + fixes |

### 🧭 Βήματα pentest ("Before Penetration Testing")

1. **Act like a regular user** — ακολούθησε το "Happy Path".
2. **Gather data** — βρες entry-points.
3. **Determine possible attacks** — με βάση τα δεδομένα.
4. **Social engineering** (αν χρειαστεί) — phishing/recon.
5. **Vulnerability Assessment** — automated tools.
6. **Provision for coverage** — σβήσε τα ίχνη σου.

<aside>
⚠️

**ΠΑΝΤΑ explicit permission** πριν κάνεις pentest — αλλιώς είναι παράνομο.

</aside>

### 👤 Hacker types & 📦 Box testing

- **Hackers:** **Black hat** (κακόβουλος) · **Gray hat** (testing, μερικές φορές self-gain) · **White hat** (ethical — βρίσκει & διορθώνει).
- **Box testing:** **Black box** (καμία γνώση κώδικα, user POV) · **Gray box** (μερική γνώση) · **White box** (πλήρης γνώση, developer POV, εξαντλητικό).

### 🐞 Οι 3 βασικές ευπάθειες (τύποι + πρόληψη)

**① XSS (Cross-Site Scripting)** — injection malicious script σε σελίδα.

- Τύποι: **Reflected** (επιστρέφει στο response) · **Stored** (αποθηκεύεται στον server) · **DOM-based** (αλλάζει το DOM στον browser).
- Πρόληψη: **CSP** headers · input validation/sanitization · απαγόρευση `<script>`/`<iframe>` · secure libs (React/Angular) · απόφυγε `eval()`/`innerHTML`.

**② SQL Injection** — malicious SQL στο input. Σπάει **CIA** (Confidentiality / Integrity / Authentication).

- Πρόληψη: **Parameterized / prepared statements** (το #1) · escaping user input · input validation · regular updates.

**③ Path Traversal** — πρόσβαση σε αρχεία εκτός επιτρεπτού (π.χ. `../`).

- Πρόληψη: directory **whitelisting** · **absolute paths** · reject `../` & `%00` · σωστά file permissions · logging/monitoring.

### 📚 Άλλα θεωρητικά (πιθανές ερωτήσεις)

- **CWE vs CVE:** **CWE** = *τύπος* αδυναμίας (π.χ. SQLi, buffer overflow) · **CVE** = *συγκεκριμένο instance* σε συγκεκριμένη έκδοση λογισμικού (π.χ. RCE στο XYZ app λόγω SQLi).
- **DOM:** αναπαριστά τη σελίδα ως nodes/objects ώστε η JS να αλλάζει δομή/στυλ/περιεχόμενο (language-agnostic "καμβάς"). **iFrame:** HTML element που ενσωματώνει άλλη σελίδα μέσα στην τρέχουσα.
- **DevSecOps:** security σε **κάθε** στάδιο dev/deploy (Security-by-Design). Best practices: εκπαίδευση χρηστών, AI security tools, regular audits, **update dependencies**, report νέων ευπαθειών.
- **ZAP Full-Scan GitHub Action:** τρέχει DAST στο CI/PR → ποστάρει ευπάθειες ως issues (automation, consistency, coverage).

## ⚡ Lab 11 — Performance Testing (k6) *[contingency]*

**Τι είναι:** κομμάτι της **δυναμικής ανάλυσης** (όπως το DAST), αλλά αντί για ασφάλεια μετράει **απόδοση** (responsiveness, stability, scalability, resource usage) υπό φόρτο. Εργαλείο lab: **k6 (Grafana)** — CLI.

<aside>
🧪

**Τι έκαναν στο εργαστήριο:** εγκατέστησαν το **k6**, έγραψαν ένα `performance-testing-lab.js` με **options (stages/thresholds)** + μια `default` συνάρτηση που χτυπάει το endpoint, και το έτρεξαν με `k6 run ...` εναντίον ενός running app ([localhost:3000](http://localhost:3000) / sgarden / Juice Shop). Έδειξαν και πώς μπαίνει στο **CI** (GitHub Action). Καθαρά demo — κανείς δεν τρέχει 5000+ VUs σε εξέταση.

</aside>

### 🔑 Μπορεί να βγει άσκηση;

- **Πρακτική (run) → απίθανο:** δεν γίνεται να τρέξεις στα αλήθεια load test σε εξέταση (χρόνος/υποδομe).
- **Θεωρία → πολύ πιθανό:** τύποι testing (load/stress/spike/soak), metrics (response time/throughput/latency...), ποιο parameter εστιάζει κάθε test.
- **Μικρή άσκηση “γράψε/συμπλήρωσε k6 script” → μέτρια πιθανό:** π.χ. «γράψε threshold p(95)<500» ή «βάλε check για status 200» — **δεν χρειάζεται εκτέλεση**, μόνο σωστή σύνταξη. Γι' αυτό κράτα το boilerplate παρακάτω.

### 🧱 Ο σκελετός ενός k6 script (3 μέρη)

```jsx
import http from 'k6/http';
import { sleep, check } from 'k6';

// 1) OPTIONS — πόσοι χρήστες & όρια επιτυχίας
export const options = {
  stages: [
    { duration: '10m', target: 5000 }, // ramp up → 5000 VUs
    { duration: '100m', target: 5000 }, // hold σταθερά
    { duration: '10m', target: 0 },     // ramp down → 0
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],   // 95% < 500ms (ΔΕΝ σπάει το test)
    http_req_failed: [{ threshold: 'rate<0.001', abortOnFail: true }], // <0.1% fails (ΣΠΑΕΙ)
  },
};

// 2) DEFAULT — τι κάνει ΚΑΘΕ virtual user (VU) σε κάθε iteration
export default () => {
  const resp = http.get('http://localhost:3000');
  // 3) CHECKS — validations (ΔΕΝ σπάνε το test, μόνο καταγράφονται)
  check(resp, { 'Is status 200?': (r) => r.status === 200 });
  sleep(Math.random() * 5); // think time 0–5s ανά VU
};
```

### 📦 k6 έννοιες-κλειδιά

| Όρος | Τι είναι |
| --- | --- |
| **VU** (virtual user) | ένας προσομοιωμένος χρήστης που τρέχει την `default` σε loop |
| **stages** | φάσεις με `duration`  • `target` VUs (ramp up / hold / ramp down) |
| **thresholds** | όρια pass/fail βάσει των Non-Functional Requirements |
| **`abortOnFail: true`** | σταματάει το test μόλις παραβιαστεί το όριο |
| **check** | validation (status/payload) — καταγράφει, ΔΕΝ ρίχνει το test |
| **sleep** | think time — προσομοιώνει ρεαλιστική παύση χρήστη |
| **scenarios / executor** | πιο advanced setup (π.χ. `ramping-vus`) αντί για σκέτο stages |

### 📊 Συχνά thresholds (copy-paste)

```jsx
http_req_duration: ['p(95)<500'],   // 95% των requests < 500ms
http_req_duration: ['p(99)<1000'],  // 99% < 1000ms
http_req_duration: ['max<2000'],    // max < 2s
http_req_failed: ['rate<0.001'],    // error rate < 0.1%
http_reqs: ['rate>100'],            // ≥ 100 requests/sec (throughput)
```

### 🎯 Τύποι Performance Test (⭐ πίνακας για θεωρία)

| Test | Σκοπός | Main parameter |
| --- | --- | --- |
| **Load** | συμπεριφορά στο **αναμενόμενο** φόρτο | Response Time |
| **Stress** | βρες το **breaking point** του server | Response Time + Throughput |
| **Spike** | απότομη αύξηση/μείωση φόρτου | Response Time |
| **Endurance / Soak** | παρατεταμένος φόρτος (ώρες) → memory leaks | Memory |
| **Volume** | βαριές queries / μεγάλα δεδομένα (DB) | Response Time of Queries |
| **Availability** | 24x7 uptime (π.χ. 99.9%) | Response Time / faults |
| **Scalability** | load test με προβλεπόμενη αύξηση χρηστών | Response Time + max TPS |

### 📏 Performance Metrics (ορισμοί)

- **Response Time:** χρόνος για να γυρίσει απάντηση ("round-trip", περιλαμβάνει latency).
- **Latency:** καθυστέρηση μετάδοσης στο δίκτυο (χωρίς χρόνο processing).
- **Throughput:** transactions/requests ανά μονάδα χρόνου (χωρητικότητα).
- **Error Rate:** ποσοστό αποτυχημένων requests.
- **Resource use:** Memory / CPU Utilization, Network Usage, Disk I/O.

**① 🔌 Prefix του endpoint** *(λάθος #1 → 404)*

- `app.js`/`server.js` → ψάξε `app.use(` → υπάρχει prefix; π.χ. `app.use('/api/v1', routes)` ή `app.use(config.apiPrefix, routes)`.

- Διπλή επιβεβαίωση: frontend `api.js`/axios → `baseURL: '.../api/v1'`.

**② 📦 Σχήμα απάντησης / wrapper** *(λάθος #2 → `body.data`)*

- Στον controller δες το return: σκέτο `res.json(x)` ή helper `ok(res, msg, x)`;

- 🔎 Σίγουρος τρόπος: `console.log(body)` μία φορά → δες το σχήμα.

**③ 🌐 Base URL του test wrapper**

- Auth-gated κουμπί: στο `*DetailsPage.jsx` ο handler κάνει `if(!isAuthenticated) navigate('/login')`.
